Azure "Project Zero": Deploying a Foundational Linux Virtual Machine

Objective:

To gain hands-on experience with core Azure Infrastructure as a Service (IaaS) components by deploying a single Linux Virtual Machine (VM) within a secure Virtual Network (VNet). This project focuses on understanding the VM creation process, network configuration basics, establishing SSH connectivity for remote management, and familiarizing myself with the Azure portal.

Cloud Platform & Services Used:

  • Cloud Provider: Microsoft Azure
  • Core Services:
    • Azure Resource Groups
    • Azure Virtual Networks (VNet) & Subnets
    • Azure Network Security Groups (NSG)
    • Azure Virtual Machines (Compute)
    • Azure Public IP Addresses
    • Azure Managed Disks

Deployment Steps & Configuration:

The deployment process involved navigating the Azure portal and configuring the necessary parameters for the virtual machine and its associated resources.

1. Prerequisite Setup (Performed prior):
  • Resource Group Creation: A Resource Group named rg-projectzero-lab was created to organize all resources for this project.
  • Virtual Network (VNet) Creation: A VNet named vnet-projectzero (e.g., with address space 10.0.0.0/16) and a default subnet (e.g., subnet-default with address space 10.0.0.0/24) were created within the rg-projectzero-lab Resource Group.
2. Virtual Machine Creation - "Basics" Tab:

The following configurations were set on the "Basics" tab of the "Create a virtual machine" wizard, as shown in the screenshot below:

  • Subscription: Azure subscription 1
  • Resource group: rg-projectzero-lab
  • Virtual machine name: vm-ubuntu-projectzero
  • Region: Brazil South
  • Availability options: No infrastructure redundancy required
  • Zone options: Self-selected zone
  • Security type: Standard
  • Image: Ubuntu Server 24.04 LTS - Gen2
  • VM architecture: x64
  • Size: Standard B1s (1 vcpu, 1 GiB memory)
  • Authentication type: SSH public key
  • Username: azureuser
  • SSH key format: RSA
  • Key pair name: vm-ubuntu-projectzero_key_0524
  • Public inbound ports: SSH
  • Azure Spot: No
3. Virtual Machine Creation - "Disks" Tab (Configuration Summary):
  • OS disk type: Standard SSD
  • OS disk size: Image default
  • Other settings were left as default.
4. Virtual Machine Creation - "Networking" Tab (Configuration Summary):
  • Virtual network: The previously created vnet-projectzero was selected.
  • Subnet: The subnet-default within vnet-projectzero was selected.
  • Public IP: A new dynamic Public IP address was automatically created and associated with the VM's Network Interface Card (NIC).
  • NIC network security group (NSG): Basic was selected. The rule to allow inbound SSH on port 22 was applied to this NSG.
  • Other settings were left as default.
5. Review + Create & Deployment:
  • After configuring all necessary tabs, the configuration was reviewed.
  • The "Validation passed" message confirmed the configuration was valid.
  • Clicked "Create" to initiate deployment.
  • The private key (.pem file) for SSH was downloaded and stored securely.

Post-Deployment & Verification:

  1. Locate Public IP: Noted the Public IP address of vm-ubuntu-projectzero from the Azure portal.
  2. SSH Connection: Connected to the VM using the downloaded private key: 
    ssh -i /path/to/your/vm-ubuntu-projectzero_key_0524.pem azureuser@<VM_PUBLIC_IP_ADDRESS>

    (Note: Path and IP address are placeholders.)

  3. Verification: Ran basic commands like uname -a and ls to confirm operational status.

Key Learnings:

  • Gained practical experience navigating the Azure portal for VM creation.
  • Understood the relationship between Resource Groups, VNets, Subnets, NSGs, and VMs.
  • Successfully configured a Linux VM for SSH access using public key authentication.
  • Reinforced the importance of secure private key management.
  • Became familiar with selecting VM sizes appropriate for free tier allowances.

Cost Management Note:

The "Estimated monthly costs" in Azure reflect potential costs if resources run 24/7 outside free tier allowances. The Standard B1s VM is eligible for 750 free hours/month for 12 months. By stopping (deallocating) the VM when not in use, compute costs are minimized. Budget alerts were also configured.

Project One: VNet Peering & Routing in Azure

Objective:

To implement and test VNet peering between two Azure Virtual Networks in the same region, configure custom User Defined Routes (UDRs) to direct traffic through a Network Virtual Appliance (NVA) or a specific gateway, and verify connectivity.

Cloud Platform & Services Used:

  • Azure Virtual Networks (VNet) & Subnets
  • Azure Virtual Machines (for test VMs and potentially an NVA like VyOS)
  • Azure VNet Peering
  • Azure Route Tables (UDRs)
  • Azure Network Security Groups (NSG)

High-Level Steps:

  1. Create two distinct VNets (e.g., VNet-A and VNet-B) with non-overlapping IP address spaces.
  2. Deploy test VMs in each VNet.
  3. Establish VNet peering between VNet-A and VNet-B.
    • Configure peering from VNet-A to VNet-B.
    • Configure peering from VNet-B to VNet-A.
  4. Test basic connectivity between VMs in the peered VNets.
  5. (Optional Advanced) Deploy a Network Virtual Appliance (NVA) like VyOS in one VNet.
  6. (Optional Advanced) Create Route Tables with UDRs to force traffic from one VNet to another through the NVA.
  7. Verify traffic flow and troubleshoot routing.

Key Learnings:

  • Practical understanding of VNet peering setup and its benefits/limitations.
  • Experience with creating and associating Route Tables.
  • Knowledge of how User Defined Routes override default system routes.
  • Troubleshooting network connectivity in a peered environment.
